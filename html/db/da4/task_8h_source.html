5 // Task is a wrapper over tbb::task to support policies.
6 //
7 // There are two kind of tasks,
8 // - <task-id, instance-id> specifies task with a given instance-id
9 // - <task-id> specifies task without any instance
10 //
11 // The policies can be specified in the form of,
12 // task(tid0) => <tid1, -1> <tid2, 2> <tid3, 3>
13 // The rule implies that,
14 // - Task <tid0, *> cannot run as long as <tid1, *> is running
15 // - Task <tid0, 2> cannot run as long as task <tid2, 2> is running
16 // - Task <tid0, 3> cannot run as long as task <tid3, 3> is running
17 //
18 // The policy rules are symmetric. That is,
19 // - Task <tid1, *> cannot run as long as <tid0, *> is running
20 // - Task <tid2, 2> cannot run as long as task <tid0, 2> is running
21 // - Task <tid3, 3> cannot run as long as task <tid0, 3> is running
22 //
23 // If task_instance == -1, means instance is not applicable.
24 // It implies that, any number of tasks with instance -1 can run at a time
25 //
26 // If task_instance != -1, only one task of given instnace can run at a time
27 //
28 // When there are multiple tasks ready to run, they are scheduled in their
29 // order of enqueue

148 // The TaskScheduler keeps track of what tasks are currently schedulable.
149 // When a task is enqueued it is added to the run queue or the pending queue
150 // depending as to whether there is a runable or pending task ahead of it
151 // that violates the mutual exclusion policies.
152 // When tasks exit the scheduler re-examines the tasks on the pending queue
153 // which may now be runnable. It is important that this process is efficient
154 // such that exit events do not scan tasks that are not waiting on a particular
155 // task id or task instance to have a 0 count.
